# LLMPET / Octopus 运行期安全、性能与体验深审计报告

**审计日期：** 2026-07-20
**审计版本：** 0.1.1
**审计基线：** 第一轮审计产物 `LLMPET-audited-2026-07-20.tar.gz`
**范围：** Electron 主/渲染进程边界、本地 HTTP 与 hook 通信、IPC、权限决策、文件与进程操作、计量持久化、长时间运行、异常网络/大数据量、跨平台打包脚本、依赖供应链、文档与测试一致性。

## 1. 结论

本轮不是单纯增加防御性判断，而是把应用的主要信任边界重新收紧：

1. 渲染器改为沙箱化、上下文隔离、无 Node 集成，并加入严格 CSP、导航/新窗口/下载/设备权限拦截和 IPC 发送者校验。
2. 回环 HTTP 从“仅监听 127.0.0.1”升级为“回环地址 + 每次启动随机令牌 + 请求来源/服务身份校验”，同时限制连接、请求头、正文、字段和慢请求。
3. 权限桥统一采用失败时询问（fail-safe `ask`），批量允许规则限定在当前会话并自动过期。
4. transcript、会话目录、配置、计价缓存和计量状态全部改为有界读取；大 JSONL 使用固定内存分块扫描，避免首次启动或长期积累后阻塞主进程。
5. 计量状态使用无原型字典和有限数值清洗，阻断动态模型名触发的原型污染以及 `Infinity`/异常大数破坏界面与持久化文件。
6. 打包启动器默认保留 Chromium 沙箱；Shell 路径严格引用；Windows 卸载 hook 以 Electron 的 Node 模式执行。
7. Electron 从 33 升级到 43.1.1，并重新锁定依赖树。

在当前 Linux、无图形界面、Electron 二进制不可下载的执行环境中，所有可执行的无头测试、压力测试、语法遍历、依赖审计、静态安全检查、发布清单检查和补丁重放验证均通过。**这显著提高了后续正常运行的置信度，但不能替代 Windows/macOS/Linux 三平台真实 GUI 与安装包验收，也不能承诺软件绝无缺陷。**

## 2. 威胁模型

### 纳入范围

- 恶意或被污染的 transcript、session 元数据、模型名、工具参数与本地配置。
- 浏览器页面或同机普通进程在不知道本次启动令牌时伪造本地状态/权限请求。
- 渲染层 XSS 后尝试访问 Node/Electron 能力、任意导航、新窗口或 IPC。
- 巨型、截断、畸形、符号链接或高速增长的本地文件。
- 残缺 HTTP 请求、并发请求、重复权限请求、异常 provider 响应。
- 显示器拔插、分辨率变化、重复启动、provider 禁用、服务启动竞态、长时间运行。

### 明确不纳入

- 已完全控制同一系统账号的恶意程序：它可能读取该账号可读的 `~/.octopus/runtime.json` 和用户会话文件。
- 已获得管理员/root 权限、注入 Electron/Node 进程或篡改已安装程序的攻击者。
- 上游 Claude Code、CodeWhale、npm registry、操作系统或 Electron 自身的未知漏洞。
- 未签名或未公证安装包在真实操作系统上的分发信任问题。

## 3. 主要发现与修复

| 严重度 | 发现 | 正常使用/安全影响 | 修复 |
|---|---|---|---|
| 高 | 渲染器未显式启用沙箱，页面缺少 CSP | 本地 HTML 注入一旦发生，影响面过大 | `sandbox/contextIsolation/webSecurity` 开；Node/webview/不安全内容关；加入严格 CSP |
| 高 | IPC handler 未校验发送页面 | 被误导航或额外 WebContents 可调用主进程能力 | 对每个 IPC 校验 sender frame 与精确本地页面允许表 |
| 高 | 本地 HTTP 只有回环绑定，没有启动身份 | 同机页面/进程可伪造状态或权限请求 | 256-bit 随机启动令牌、常量时间比较、Host/Origin/Referer/服务身份校验 |
| 高 | Linux/Windows 启动器默认 `--no-sandbox` | 发布包抵消渲染器沙箱 | 默认不传；仅显式诊断变量 `OCTOPUS_DISABLE_CHROMIUM_SANDBOX=1` 才降级 |
| 高 | 大 transcript 整段读入内存；会话列表最坏可同步解析数 GB | 首次运行、长期使用或损坏文件可卡死/OOM | 4 MiB 分块 JSONL、每轮 32 MiB 总预算、文件/会话/候选数量硬上限、轮转扫描 |
| 高 | 动态模型键写入普通对象；token/cost 可成为非有限值 | 原型污染、状态损坏、面板渲染异常 | 无原型字典、危险键拒绝、有限/有界数值、集合和状态文件上限 |
| 高 | provider 禁用后旧 hook 可能继续工作；服务绑定慢时首次安装可能永久漏掉 | 用户以为已禁用仍收到事件，或首次启动无功能 | 启动时按配置对账安装/卸载；服务就绪重试最多 10 秒 |
| 中 | 残缺 HTTP 正文可长期占连接 | 本地服务卡顿并影响后续正常请求 | 10 秒请求超时、连接/请求头/keep-alive 限制、关闭时销毁连接 |
| 中 | 权限队列、重复请求、会话集合无充分上限 | 长时间运行内存增长、重复弹窗 | 队列/重复键/会话数量硬上限；会话结束和关闭时清理 |
| 中 | CodeWhale “始终允许”跨会话存在 | 一次授权可能影响无关任务 | 仅当前 session 生效，30 分钟滑动过期，SessionEnd/退出/禁用时清除 |
| 中 | provider 不可达、返回伪响应或勿扰路径语义不统一 | 极端情况下过度放行或 CLI 卡住 | 所有失败/超时/非法响应统一返回 `ask`，只发送一次响应 |
| 中 | transcript 路径、配置/缓存读取缺少统一边界 | 误读任意文件、符号链接绕过、启动卡顿 | 目录包含校验、拒绝符号链接、常规文件检查、单文件/总预算上限 |
| 中 | Shell 命令和路径引用不完整；Unix 把 `command` 当可执行文件 | 路径带空格时启动失败，特殊字符可能被误解释，PATH 探测失效 | POSIX/PowerShell 严格引用；通过 shell 执行 `command -v` |
| 中 | 计价别名在基础价格缺失时仍可注册 | 个别模型可能绕过默认价格回退，低估花费 | 仅在基础定价存在时注册别名，新增回归测试 |
| 中 | Windows 卸载脚本直接调用 Electron 可启动 GUI 而非 Node 脚本 | 卸载后残留 hook 或误启动应用 | 设置 `ELECTRON_RUN_AS_NODE=1` 后执行卸载器 |
| 低 | 统计循环可重叠，日志同步写入，transcript 未变化仍重复解析 | 慢磁盘/大量会话时周期性卡顿 | 单次 timeout 调度、异步有界日志队列、mtime/size 未变则复用缓存 |
| 低 | 显示器拔插后窗口可能离屏 | 桌宠/面板“消失” | display add/remove/metrics 事件中重新夹取到可见工作区 |
| 低 | 状态/计价缓存直接覆盖，权限默认继承系统 umask | 崩溃时截断，多用户机器上元数据暴露 | 临时文件 + rename 原子写；目录 0700、敏感文件 0600 |

## 4. 关键设计说明

### 4.1 本地随机令牌不是系统级沙箱

应用将本次启动的令牌写入 `~/.octopus/runtime.json`，hook 安装器读取并注入受控请求。它能阻止不知道令牌的网页、误配置客户端和普通同机调用，但不能抵御已经完全控制同一用户账号的恶意进程。因此文档不再把 loopback 描述为“天然可信”。

### 4.2 权限失败策略

授权桥的目标不是在桌宠不可用时替用户做决定，而是保证 CLI 不因异常响应而获得更宽权限。所有身份不匹配、畸形 JSON、超时、队列超限、会话缺失和服务关闭路径均回退到 `ask`。

### 4.3 有界计量

Claude JSONL 计量采用固定 4 MiB 块并保留跨块尾部；单轮全局读取预算为 32 MiB。超长单行会被跳过并推进游标，避免永久卡在同一偏移。CodeWhale 会话扫描限制候选数量、单文件大小、总同步读取预算和最终返回数量。

### 4.4 Electron 升级

锁文件已更新到 Electron 43.1.1。源代码未使用本轮大版本中已删除或明显变化的 API，现有主进程、preload 和静态适配测试均通过；但真正的 Chromium/Node 二进制未能在本环境下载，所以仍需在目标平台执行 GUI 启动与打包验收。

## 6. 尚未完成的发布级门禁

1. **真实 Electron GUI：** 本环境无法下载 Electron 平台二进制，因此未执行窗口创建、托盘、拖拽、显示器事件、原生焦点辅助程序的真实运行。
2. **三平台产物：** 没有在 macOS/Windows/Linux 分别构建并启动最终安装包；当前只验证了脚本语法、清单、依赖和启动参数。
3. **签名与公证：** Windows Authenticode、macOS codesign/notarization 尚未配置。
4. **自动更新：** 若未来加入 updater，应额外验证签名、降级保护、原子替换和运行中 ASAR 更新行为。
5. **`file://`：** 当前页面仍通过精确允许表、CSP 和 IPC sender 校验保护；更进一步可迁移到自定义协议，但应在真实 Electron 环境完成兼容验证后再发布。
6. **Electron fuses：** 生产包可评估关闭不需要的 fuses；当前 Windows hook 卸载依赖 `ELECTRON_RUN_AS_NODE`，不能在未重构卸载链路前直接关闭 RunAsNode。

## 7. 目标平台发布验收步骤

按以下顺序执行，任一步失败都不应发布：

```bash
npm ci
npm run test:all
npm audit
npm start
```

GUI 手工检查至少包括：

- Claude 与 CodeWhale 各完成一次完整会话、授权允许/拒绝/超时/服务退出路径；
- 启用/禁用 provider 后检查 hook 是否立即对账；
- 断网、CLI 不存在、transcript 被截断/快速增长时应用不崩溃；
- 多显示器拔插、缩放比例变化、休眠唤醒后窗口仍可见；
- 面板会话/计量连续运行数小时，主进程内存无持续单调增长；
- 打包产物中的 provider、hook、资源、生产依赖均可读取；
- Windows/macOS 卸载后用户配置保留，但 Octopus 自有 hook 无残留；
- 默认启动参数中不存在 `--no-sandbox`。

随后分别执行：

```bash
npm run package:linux
npm run package:win
npm run package:mac
```

每个平台必须在干净用户账号或虚拟机中完成首次安装、首次启动、退出、重启、卸载和二次安装。

## 8. 维护建议

- 每次升级 Electron 或 provider CLI，至少运行 `npm ci && npm run test:all && npm audit`，并重做真实 GUI 冒烟。
- 不要把 runtime token、usage 状态、用户 transcript 或安装后的 `node_modules` 提交到仓库或源码包。
- 新增 IPC 时必须同步加入 sender 校验；新增 HTTP route 时必须声明 token、body、字段、超时和失败语义。
- 新增动态键或持久化字段时使用无原型字典和有限数值清洗，不要直接信任 transcript。
- 任何为了“兼容 Linux”而加入 `--no-sandbox` 的改动都必须被视为安全降级，并保持显式、临时、可审计。
